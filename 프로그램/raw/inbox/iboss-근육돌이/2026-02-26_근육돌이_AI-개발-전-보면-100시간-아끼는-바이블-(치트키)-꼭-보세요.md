# AI 개발 전 보면 100시간 아끼는 바이블 (치트키) 꼭 보세요

안녕하세요 근육돌이입니다.

오늘은 저희 CCFM(콘크리트파머스)에서 실제로 사용하고 있는 바이브코딩 방법론을 전부 공개합니다.

솔직히 말씀드리면, 이거 완성하기까지 삽질을 엄청나게 했습니다. 설계 없이 코딩 시작했다가 3시간 날리고, 파일명 하나 잘못 바꿨다가 연쇄 오류 터지고, SPEC.md 300줄 써놨더니 AI가 중간 내용 통째로 누락하고... 진짜 별별 일이 다 있었습니다. ㅎㅎ

근데 이 삽질들 덕분에 지금은 이 완성됐습니다.

이 방법론, 이름이 좀 거창하게 들릴 수 있는데 결론부터 말씀드리면 이렇습니다.

**"MD****파일로 설계하고, AI가 코드를 짜고, 사람은 관리만 한다."**

코드를 몰라도 됩니다. 개발자가 아니어도 됩니다. 한국어로 AI한테 말만 할 수 있으면 충분합니다.

이 글에서 다룰 내용은 총 두 파트입니다. 파트 1은 방법론 그 자체입니다. 설계부터 개발, 수정, 운영까지 전체 워크플로우를 다룹니다. 파트 2는 실전 시행착오 10가지 케이스입니다. 저희가 실제로 이렇게 했다가 망하고, 이렇게 해결한 사례들을 케이스별로 정리했습니다.

두 파트 합쳐서 꽤 긴 글입니다. 목차 보시고 필요한 부분만 골라 읽으셔도 됩니다. 자~ 그럼 바로 시작해볼게요!

#### **파트 1. 바이브코딩 방법론 - 설계부터 운영까지**

##### **1.****핵심 철학: 설계가 90%, 코드는 출력물이다**

바이브코딩을 처음 접하는 분들이 가장 많이 하는 실수가 있습니다. AI한테 바로 "이거 만들어줘"라고 하는 겁니다. 그러면 AI가 뭔가를 만들어주긴 합니다. 근데 막상 써보면 내가 원하던 게 아니거나, 수정하려고 하면 다른 부분이 깨지거나, 나중에 유지보수가 안 되는 상황이 옵니다.

저희가 이 방법론을 만들면서 가장 먼저 깨달은 게 바로 이겁니다. AI 코딩에서 결과물의 품질을 결정하는 건 코드 실력이 아닙니다. 설계 능력입니다. 아무리 AI가 코드를 잘 짜줘도, 설계가 엉망이면 결과물도 엉망입니다. 반대로 설계만 제대로 되어있으면, 코드를 전혀 몰라도 원하는 결과물이 나옵니다.

그래서 저희 방법론의 핵심 철학은 딱 하나입니다.

이게 무슨 말이냐면, 코드는 AI가 짜주니까 언제든 다시 생성할 수 있습니다. 근데 설계 문서인 MD 파일은 사람이 만들어야 합니다. 그래서 진짜 공들여야 할 게 MD 파일이고, 코드는 MD 기반으로 AI가 뽑아내는 출력물로 취급하면 됩니다.

이 철학이 자리잡히면 개발 방식이 완전히 달라집니다. 수정할 일이 생겼을 때 코드 들여다보면서 어디 고치나 고민하는 게 아니라, MD 먼저 수정하고 코드 재생성하면 됩니다. 코드가 완전히 망가져도 MD만 살아있으면 30분이면 복구됩니다.

이게 비개발자가 AI 코딩을 할 수 있는 이유이자, 이 방법론의 출발점입니다.

#### **2. MECE****원칙 - 코드 설계의 뼈대**

MECE라는 말, 컨설팅 하시는 분들은 익숙하실 겁니다. Mutually Exclusive, Collectively Exhaustive. 한국어로 하면 "겹치지 않게, 빠짐없이"입니다. 맥킨지에서 나온 개념인데, 저희가 이걸 코드 설계에 그대로 적용했습니다.

근데 사실 코딩을 전혀 몰라도 이 개념은 직관적으로 이해됩니다. 팀을 짤 때를 생각해보세요. 영업팀이 마케팅 일도 하고, 마케팅팀이 영업 일도 하면 어떻게 됩니까? 책임 소재가 불분명해지고, 충돌이 생기고, 나중에 뭔가 잘못됐을 때 어디서 문제가 생긴 건지 파악이 안 됩니다. 코드도 똑같습니다.

 - 각 Step과 그룹이 자기 영역만 담당합니다. step1은 크롤링만, step2는 필터링만, step3은 리포트만. 다른 폴더는 절대 건드리지 않습니다. 이게 지켜지면 나중에 step2에 문제가 생겼을 때 step2만 보면 됩니다.

 - 설계도와 hook 파일로 모든 Step과 그룹이 연결됩니다. 빈 구간이 없습니다. 데이터가 step1에서 나와서 step2로 가고, step2에서 나와서 step3으로 가는 전체 흐름이 끊김 없이 연결됩니다.

이 두 가지가 지켜지면 엄청난 일이 가능해집니다.  연결 인터페이스, 즉 입출력 파일명만 맞으면 됩니다.

##### **3. MD****라이프사이클 - MD로 시작해서 MD로 끝난다**

저희 방법론에서 MD 파일이 얼마나 중요한지, 전체 워크플로우를 보시면 바로 이해됩니다.

① MD 설계 → ② 코드 생성 → ③ 수정 필요 시 MD 먼저 수정 → ④ 코드 재생성 → ⑤ 완료 후 MD 업데이트 요청

 이게 순환 구조입니다.

왜 이렇게 하냐고요? MD 파일은 한국어로 씁니다. 개발자가 아니어도 읽고 수정할 수 있습니다. 반면 코드는 Python이든 JavaScript든 개발 언어로 되어있어서 비개발자가 직접 수정하기 어렵습니다. 그러니까 비개발자가 관리할 수 있는 MD를 진짜 자산으로 유지하고, 코드는 AI가 MD 보고 자동으로 뽑아내는 구조를 만드는 겁니다.

 MD 먼저 수정하고, 그 다음에 코드 재생성입니다. 코드를 직접 건드리면 MD와 코드가 싱크가 안 맞아집니다.

그래서 수정의 마지막 단계는 항상 이겁니다.

지금 최신 코드 바탕으로 확인해서 계획서랑 hook, md파일 최신화 해줘

Claude Code한테 이 한 마디면 됩니다. Leader Agent가 전체 MD 문서를 최신화해줍니다. MD로 시작해서 MD로 끝나는 순환이 완성됩니다.

##### **4. 5****계층 파일 구조 - 건물 짓듯 설계하라**

저희 방법론에서 모든 프로젝트는 5가지 파일로 구성됩니다. 건물로 비유하면 이렇습니다.

 - 건물 설계 도면입니다. 전체 프로젝트 구조와 목적을 담습니다. AI와 사람 모두 읽습니다.

 - 배관, 배선 설계입니다. Step과 그룹 간 연결 규칙을 정의합니다.

 - 시공 상세도입니다. 각 모듈의 개발 명세서로 AI 개발용입니다. 4,500자(100줄) 이내로 작성합니다.

 - 군사 매뉴얼입니다. "Claude Code한테 뭐라고 말하면 되는지" 알려주는 한국어 가이드입니다.

 - 운영 매뉴얼입니다. 수정 범위별 판단 기준과 의존성 맵이 들어있습니다.

이 5가지가 다 갖춰진 프로젝트와 그냥 코드만 있는 프로젝트는 6개월 후 유지보수 난이도가 완전히 다릅니다. 코드만 있으면 원래 만든 사람만 손댈 수 있습니다.

##### **5.****소규모 vs 대규모 프로젝트 구조 차이**

기준은 단순합니다.

소규모는 구조가 단순합니다. 프로젝트 폴더 하나 안에 설계도.md, 수정가이드.md, 수정전략.md, hook 폴더, 그리고 step1부터 stepN 폴더가 나란히 들어갑니다. 각 step 폴더 안에는 SPEC.md와 실제 코드가 같이 있습니다.

대규모는 피라미드 구조로 확장됩니다. Step이 10개가 넘으면  예를 들어 Step이 100개라면, 7개씩 묶어서 그룹 약 15개를 만듭니다.

그룹 최적 사이즈는 6개-7개 가 최적입니다. 3개 이하면 다른 그룹과 합치고, 10개 넘으면 그룹을 다시 나눠야 합니다. 1그룹 AI 읽는 총량은 3만자 이내로 유지합니다.

 10,000개 Step도 피라미드 4단이면 커버됩니다. 토스 같은 대형 앱도 이 구조입니다. 100명 개발자 팀 = 100개 AI Agent 팀입니다.

 소규모로 시작하고, Step이 10개 넘을 때 그룹화하면 됩니다.

##### **6.****설계의 핵심 - 입력/출력 정의**

5계층 구조를 갖췄다고 설계가 끝난 게 아닙니다. 설계에서 가장 중요한 작업이 남아있습니다. 바로 각 Step의 입력과 출력을 정의하는 겁니다.

왜 이게 핵심이냐? 간단합니다. Step들이 서로 데이터를 주고받아야 하는데, 주는 쪽이 JSON으로 보내고 받는 쪽이 CSV를 기대하면 연결이 안 됩니다.

실제로 SPEC.md에 입출력을 어떻게 정의하는지 보여드리겠습니다.

•          step1 출력: raw\_data.json (JSON 배열, 필드: title, price, url)

•          step2 입력: raw\_data.json → 출력: filtered\_data.json

•          step3 입력: filtered\_data.json → 출력: report.html

이렇게 딱 써놓으면 AI가 각 Step을 개발할 때 앞뒤 Step이 뭘 주고 뭘 받는지 정확히 알고 구현합니다. 연결 오류가 0이 됩니다.

 우리가 받고 싶은 결과물의 예시를 주면 AI가 역설계해서 그대로 만들어줍니다. 예시 없이 개발하면 AI가 자기 기준으로 결과물 형태를 결정합니다.

##### **7. SPEC.md****작성법 - AI의 시공 상세도**

SPEC.md는 5가지 구성요소로 이루어집니다.

이 모듈이 뭘 하는지 한눈에 알 수 있게 씁니다.

뭘 받고 뭘 내보내는지, 파일명과 데이터 형식까지 명시합니다.

중요도 별 다섯 개입니다.

처리 단계와 조건문을 씁니다.

입니다. 앞뒤 Step과의 관계, 파일명을 씁니다. 중요도 별 다섯 개입니다.

사용할 라이브러리, 에러 처리 방식, 생성되는 파일 목록입니다.

 이보다 길어지면 Step을 더 쪼개야 한다는 신호입니다.

SPEC.md 작성이 처음이라면 이렇게 시작하세요. Claude한테 "이런 기능을 하는 모듈의 SPEC.md 초안 만들어줘"라고 요청하면 됩니다. AI가 초안을 만들어주면 그걸 검토하고 수정하는 방식으로 작성하면 훨씬 쉽습니다. ㅎㅎ

##### **8.****코드 수정 전략 - 삭제 후 재작성이 답이다**

자~ 이제 많은 분들이 가장 많이 실수하는 부분입니다. 코드 수정입니다.

Claude Code한테 "여기 수정해줘"라고 하면 어떻게 될까요? AI는 기존 코드를 삭제하지 않고 위에 새 코드를 덧붙입니다. 그러면 이전 코드가 계속 살아있으면서 새 코드와 충돌합니다. 이중 실행, 이중 필터링, 예상치 못한 결과물이 생깁니다.

수정 범위에 따라 전략이 달라집니다.

•          1개 Step 내부만 바꿀 때: 해당 Step 코드만 삭제 후 재작성

•          2개 Step 연결 부분이 바뀔 때: 두 Step 코드 삭제 + hook 수정

•          3개 이상 연쇄 수정: 전면 재시작

 SPEC.md 수정하고 코드 재생성하는 데 30분이면 됩니다. 부분 수정 고집하다가 4~5시간 날리는 게 훨씬 손해입니다.

나쁜 요청 방식: "step2 필터 조건 바꿔줘"

좋은 요청 방식: "step2/filter.py의 기존 필터 로직 전부 삭제하고, 새로운 필터 조건(price > 10000)으로 다시 작성해줘. 입력은 step1/raw\_data.json 그대로, 출력은 step2/filtered\_data.json 형식 유지."

 이 두 가지가 수정 프롬프트의 핵심입니다.

##### **9.****그룹 수정법 A-B-C - MECE 덕분에 가능한 것**

대규모 프로젝트에서 그룹 하나를 수정해야 할 때, 가장 많이 하는 실수가 있습니다. Claude Code한테 "전체 프로젝트 수정해줘"라고 하는 겁니다. 그러면 AI가 멀쩡한 A 그룹, C 그룹까지 건드려서 오히려 더 망가집니다.

3단계 프로세스는 이렇습니다.

입니다. 설계도-B.md, hook-B.md, SPEC.md들을 수정합니다. 전부 한국어라 어렵지 않습니다.

합니다. 코드만 삭제하고 MD는 남깁니다. "SPEC.md 기준으로 새로 짜줘"라고 하면 됩니다.

합니다. A는 마지막 Step 출력부만, C는 첫 Step 입력부만 확인합니다. hook-master.md에서 A→B, B→C 연결 업데이트합니다.

 MECE 설계가 되어있으면 B 내부가 아무리 바뀌어도 A, C 내부는 안전합니다.

##### **10.****멀티 에이전트 병렬 개발 - 4배 빠르게 만드는 법**

혼자 Claude Code 하나 켜놓고 step1, step2, step3 순서대로 개발하는 분들 많으시죠? 이것도 물론 됩니다. 근데 4배 느립니다. ㅎㅎ

•          Agent1 탭: "step1/SPEC.md 읽고 구현해" → 데이터 수집 담당

•          Agent2 탭: "step2/SPEC.md 읽고 구현해" → 데이터 가공 담당

•          Agent3 탭: "step3/SPEC.md 읽고 구현해" → AI 분석 담당

•          Agent4 탭: "step4/SPEC.md 읽고 구현해" → 결과 출력 담당

•          Leader 탭: "설계도 기준 전체 검토 + 연결" → 통합 검토 담당

4개 Agent가 동시에 각자 담당 Step을 개발하고, Leader가 전체를 통합합니다. 순서대로 개발하면 4시간 걸릴 것이 1시간이면 됩니다.

으로 역할을 나누면 최적입니다. 처음에는 Agent 2개부터 시작하세요. Leader 하나, Worker 하나. 이것만 해도 개발 속도가 눈에 띄게 빨라집니다.

##### **11.****생성형 AI 콘텐츠 개발 주의사항 - 모델 명시가 전부다**

이미지 자동 생성, 영상 자동화, TTS 음성 생성... 이런 기능을 자동화 툴에 붙일 때 정말 많이 실수하는 게 있습니다. 모델을 지정 안 하는 겁니다.

Claude Code한테 "이미지 생성 기능 만들어줘"라고만 하면? AI가 무료 오픈소스 모델을 자동으로 선택합니다. 생성된 이미지 품질이 낮아서 사용 불가 수준이 됩니다.

•          이미지 생성: DALL-E 3, Midjourney 등 모델명 명시

•          영상 생성: Runway Gen-3, Pika 등 모델명 명시

•          음성/TTS: ElevenLabs, OpenAI TTS 등 명시

•          텍스트 생성: Claude Sonnet, GPT-4o 등 명시

도 인지해야 합니다. API 가이드에서 temperature, top\_p 같은 미세 조정 파라미터를 확인하고 SPEC.md에 명시해두면 결과물 품질이 한 단계 올라갑니다.

##### **12.****수정 가이드 작성법 - 비개발자를 위한 프롬프트 매뉴얼**

코드를 몰라도 됩니다. 한국어로 요청하면 AI가 처리합니다. 비개발자가 이 문서만 보고 수정 요청할 수 있어야 합니다. 그게 수정 가이드의 완성도를 평가하는 기준입니다.

수정 가이드에 들어가야 할 내용은 6가지입니다.

•          전체 구조 요약 - 각 Step과 그룹이 뭘 하는지 한 줄씩

•          자주 수정하는 것 - 크롤링 URL, 필터 조건 등 변경 빈도 높은 설정값

•          수정 방법 예시 - Claude Code에게 실제로 할 말을 프롬프트 예시로

•          건드리면 안 되는 것 - 수정 금지 파일 목록

•          구조 변경 방법 - Step/그룹 추가·삭제 요청 방법

•          수정 전략 요약 - 몇 개 Step 변경 시 어떤 전략 쓰는지 판단 기준

 수정 완료 후 "현재 상태 기준으로 수정가이드 업데이트해줘" 요청하면 AI가 알아서 최신화해줍니다.

##### **13.****토큰/컨텍스트 관리 - AI가 잘 일하게 만드는 법**

AI가 코드를 잘 짜려면 AI가 읽는 문서가 적절한 분량이어야 합니다. 너무 길면 AI가 중간 내용을 압축하거나 누락합니다.

입니다. AI는 앞뒤를 먼저 처리하고 중간이 약해지는 특성이 있습니다. 100줄 넘으면 중간 조건문이 누락됩니다.

입니다. 설계도, hook, SPEC.md 전부 합쳐서입니다. 3만자가 넘으면 그룹을 더 나눠야 한다는 신호입니다.

 씁니다. 요약만 씁니다. 세부 내용은 각 SPEC.md에 있습니다.

 씁니다. 긴 프롬프트는 결과 압축을 유발합니다.

##### **14.****종합 체크리스트 - 프로젝트 시작부터 완료까지**

지금까지 13가지 챕터를 다 읽으셨다면, 실제로 어떻게 적용하는지 체크리스트로 정리해드립니다. 프로젝트할 때 이걸 옆에 두고 확인하면서 진행하세요.

**프로젝트 시작 전**

•          각 Step별 SPEC.md 작성 (4,500자/100줄 이내)

**개발 중**

•          각 Agent는 자기 폴더에서만 작업 (MECE 원칙)

•          수정 시 기존 코드 삭제 먼저 요청

•          3개 이상 연쇄 수정 감지 시 전면 재시작 고려

•          그룹 수정 시 A-B-C 규칙 적용 (B만 재생성, A/C 연결부만)

•          파일명 변경 시 앞뒤 Step + hook 동시 업데이트

**개발 완료 후**

•          수정가이드.md 작성 완료

•          Leader 전체 통합 테스트

•          Leader한테 "MD 전체 업데이트" 요청

•          비개발자가 수정가이드만 보고 수정 가능한지 테스트

 비개발자가 수정가이드만 보고 수정 요청할 수 있어야 진짜 완성입니다.

#### **파트 2. 실전 시행착오 10 케이스**

자~ 파트 1에서 방법론을 다 설명드렸습니다. 근데 솔직히 말씀드리면, 저희도 처음부터 이 방법론대로 한 게 아닙니다. 다 틀려보고 나서 만들어진 겁니다.

파트 2는 저희가 실제로 겪은 시행착오 10가지 케이스입니다. 각 케이스마다 어떤 상황이었는지, 왜 문제가 생겼는지, 어떻게 해결했는지, 그리고 핵심 교훈을 정리했습니다.

이미 파트 1을 읽으신 분들은 "아, 그래서 그런 원칙이 생긴 거구나"라고 연결되면서 읽히실 겁니다. 파트 1 건너뛰고 여기부터 읽으시는 분들은 각 케이스 교훈만 챙겨가셔도 충분합니다.

이게 파트 2의 핵심입니다. 여러분들은 저희가 한 실수를 안 하셔도 됩니다. 바로 시작합니다.

##### **케이스 1. 설계 없이 바로 코딩 시작 → 3시간 날림**

관련 방법론: 핵심 철학 (설계가 90%)

어떤 상황이었냐면, 자동화 툴을 하나 만들어야 하는데 머릿속에 대충 그림이 그려졌습니다. 크롤링하고, 필터링하고, 리포트 만들면 되는 거잖아요. 그래서 바로 Claude Code한테 요청했습니다.

AI가 한번에 전체 코드를 뚝딱 만들어줬습니다. 처음엔 "오 빠르다"싶었습니다. 근데 실행해보니까 크롤링 결과 형식과 필터 입력 형식이 안 맞아서 바로 오류가 떴습니다. 수정하려고 코드 들여다보는데 어디가 문제인지 파악도 어렵고, 한 군데 고치면 다른 부분이 깨지는 악순환이 시작됐습니다.

두 번째 시도는 달랐습니다. 설계도.md로 전체 구조를 먼저 정의했습니다. step1(크롤링), step2(필터), step3(리포트)로 분리하고, 각 step별 SPEC.md에 입력/출력 파일명과 형식을 명확히 정의했습니다. hook 파일로 step 간 연결 규칙을 문서화했습니다. 그러고 나서 개발 시작했더니 30분 만에 완성됐습니다.

 설계 없이 코딩을 시작하면 나중에 3~5배 시간을 더 씁니다. 설계 30분 투자가 3시간 삽질을 방지합니다. AI는 명확한 구조가 있을 때 정확하게 작업합니다.

##### **케이스 2. 입출력 정의 없이 개발 → 파일 형식 지옥**

관련 방법론: 설계의 핵심 (입력/출력 정의)

설계도는 만들었습니다. Step도 나눴습니다. 근데 각 Step의 입출력 형식을 정의 안 했습니다. "어차피 AI가 알아서 맞추겠지"라고 생각했습니다.

step1에서 크롤링 결과를 JSON으로 저장했는데, step2에서는 CSV로 읽으려고 했습니다. 파일 형식이 안 맞아서 step2가 작동을 안 합니다. step1을 CSV로 바꿨더니 이번엔 step3에서 오류가 납니다. 파일명도 step1은 output.json, step2는 data.csv, step3은 result.json으로 제각각입니다. 어디서 뭘 읽고 쓰는지 전체가 꼬여버렸습니다.

해결 방법은 SPEC.md에 입출력을 명확히 정의하는 것이었습니다.

•          step1 출력: raw\_data.json (JSON 배열, 필드: title, price, url)

•          step2 입력: raw\_data.json → 출력: filtered\_data.json

•          step3 입력: filtered\_data.json → 출력: report.html

파일명을 통일하고 형식을 명시하니까 연결 오류가 완전히 사라졌습니다.

 입출력을 정의하지 않으면 AI가 제각각 파일 형식을 선택합니다. 파일명과 데이터 형식을 SPEC.md에 명시하면 연결 오류가 0이 됩니다. 파일명이 곧 Step 간 계약입니다.

##### **케이스 3. 기존 코드 위에 덧붙여서 수정 → 이중 충돌**

관련 방법론: 코드 수정 전략 (삭제 후 재작성)

step2의 필터 조건을 바꿔야 했습니다. Claude Code한테 이렇게 요청했습니다.

AI가 새 코드를 추가했습니다. 근데 기존 필터 로직을 삭제하지 않았습니다. 결과가 어떻게 됐냐면, 기존 필터가 먼저 실행되고 새 필터가 그 위에 또 실행됐습니다. 데이터가 이중 필터링되어 결과값이 완전히 틀려버렸습니다. 디버깅에 2시간을 소비했습니다.

올바른 요청 방식은 이렇습니다.

 "수정해줘"라고 하면 AI는 기존 코드 위에 덧붙입니다. 반드시 "기존 삭제 후 새로 작성"으로 요청해야 합니다. 입출력도 함께 명시해야 앞뒤 Step 연결이 안 무너집니다.

**케이스 4. 파일명 하나 바꿨다가 앞뒤 Step 전부 오류**

관련 방법론: 파일명 = Step 간 계약

step2의 출력 파일명을 filtered\_data.json에서 clean\_data.json으로 변경했습니다. step2 코드만 수정하고 "끝났다"했습니다.

그런데 step3을 실행하니까 바로 오류가 납니다. "File not found: filtered\_data.json" step3은 여전히 filtered\_data.json을 읽으려고 하는 겁니다. step3 수정했더니 이번엔 step4에서 또 오류. 파일명 하나 바꾼 게 연쇄 오류로 번졌습니다.

올바른 요청 방식은 이렇습니다.

 파일명은 Step 간 계약입니다. 하나 바꾸면 그 파일을 쓰는 모든 Step을 확인해야 합니다. 파일명 변경 시 "이 파일 쓰는 다른 Step도 업데이트"를 반드시 포함해야 합니다.

##### **케이스 5. 3개 Step 연쇄 수정을 부분 수정으로 시도 → 4시간 낭비**

관련 방법론: 코드 수정 전략 (수정 범위별 전략)

step1, step2, step3의 로직을 모두 변경해야 하는 상황이 왔습니다. "하나씩 고치면 되겠지"라고 생각했습니다. step1부터 수정을 시작했습니다.

step1 수정 완료 → step2에서 오류 → step2 수정 → step3에서 오류 → step3 수정 → 그랬더니 step3 수정 내용이 step1 출력과 다시 안 맞아서 step1 다시 수정. 앞뒤로 왔다갔다를 반복했습니다. 결국 4시간을 썼는데 제대로 작동하지 않았습니다.

결론은 전면 재시작이었습니다.

•          ① SPEC.md 3개를 먼저 전부 수정 (한국어라 30분이면 됩니다)

•          ② step1~3 코드를 전부 삭제 (MD는 그대로 남김)

•          ③ 멀티 에이전트 3개로 동시 재생성

•          ④ Leader가 연결 확인

 3개 이상 변경이면 부분 수정 시도하지 말고 바로 전면 재시작해야 합니다. MD만 수정하고 코드 밀고 새로 짜는 게 디버깅보다 10배 빠릅니다. 코드는 버려도 됩니다. MD만 살아있으면 언제든 재생성됩니다.

##### **케이스 6. SPEC.md 300줄 작성 → AI가 중간 내용 누락**

관련 방법론: 토큰/컨텍스트 관리

복잡한 모듈을 개발해야 했습니다. 처리해야 할 조건이 많고, 로직도 복잡했습니다. 그래서 SPEC.md를 최대한 상세하게 썼습니다. 300줄짜리 SPEC.md가 완성됐습니다. "이 정도면 AI가 완벽하게 만들겠지"라고 생각했습니다.

결과물을 받아보니 앞부분 80줄은 정확하게 구현됐습니다. 근데 중간 부분 120~200줄 구간의 조건문들이 통째로 누락됐습니다. 뒷부분은 점점 압축되고 요약되어 품질이 눈에 띄게 떨어졌습니다.

해결 방법은 분리였습니다. 300줄짜리 모듈을 3개 Step으로 나눴습니다.

•          step-A: 데이터 전처리 (100줄 SPEC.md)

•          step-B: 핵심 로직 처리 (100줄 SPEC.md)

•          step-C: 결과 후처리 (100줄 SPEC.md)

각각 AI가 한번에 소화할 수 있는 양이 됐습니다. 결과물 품질이 세 파트 모두 균일하게 나왔습니다.

 SPEC.md가 100줄을 넘으면 쪼개야 한다는 신호입니다. AI는 앞뒤를 우선 처리하고 중간이 약해집니다. 4,500자로 설명 못 할 코딩은 없습니다. 분해만 잘하면 됩니다.

##### **케이스 7. 생성형 AI 모델 미지정 → 저품질 결과물**

관련 방법론: 생성형 AI 콘텐츠 개발 주의사항

영상 자동화 툴을 개발했습니다. 이미지 생성, TTS 음성, 영상 합성까지 한번에 처리하는 꽤 복잡한 툴이었습니다. SPEC.md도 잘 썼고 설계도도 갖췄습니다. 근데 모델 지정을 깜빡했습니다. SPEC.md에 이렇게만 써놨습니다.

Claude Code가 각 기능을 무료 오픈소스 모델로 자동 선택해서 구현했습니다. 이미지 품질이 너무 낮아서 사용 불가 수준이었습니다. TTS 음성은 부자연스러워서 듣기가 불편했습니다. 결과물 전체가 비전문적이라 처음부터 다시 개발해야 했습니다.

두 번째 개발에서는 SPEC.md에 모델과 API를 명시적으로 지정했습니다.

•          이미지: DALL-E 3 API 사용, 해상도 1024x1024

•          음성: ElevenLabs API, 남성 전문적 톤, stability 0.75

•          영상: Runway Gen-3 Alpha API, 4초, 720p

•          API 가이드에서 temperature, top\_p 파라미터 확인 후 명시

결과물 품질이 극적으로 향상됐습니다. 첫 번째 결과물과 비교하면 같은 툴이 맞나 싶을 정도였습니다.

 AI에게 "생성해"라고만 하면 무료 저품질 모델을 자동 선택합니다. 모델명, API, 파라미터까지 SPEC.md에 명시해야 품질이 보장됩니다.

##### **케이스 8. 아웃풋 예시 없이 개발 → 5번 재작업**

관련 방법론: 설계의 핵심 (아웃풋 스펙 + 예시 결과물)

리포트 생성 모듈을 개발했습니다. SPEC.md에 목적, 입출력, 핵심 로직 다 썼습니다. 근데 예시 결과물을 첨부하지 않았습니다.

AI가 자기 기준으로 리포트 포맷을 결정해서 만들어줬습니다. 결과물을 보니 원하는 것과 전혀 달랐습니다. 수정 요청했습니다. 또 달랐습니다. 또 수정. 이걸 5번 반복했습니다.

해결 방법은 예시를 주는 것이었습니다. SPEC.md에 이렇게 추가했습니다.

•          해상도: 1080p, 스타일: 다크모드, 폰트: 14pt

•          섹션 순서: 요약 → 주요 지표 → 차트 → 결론

•          참고 리포트 파일 첨부

AI가 예시를 역설계해서 정확히 그대로 만들어줬습니다. 수정 한 번 없이 첫 번째 결과물이 바로 원하는 형태로 나왔습니다.

 AI에게 예시를 주면 역설계해서 그대로 만들어줍니다. 원하는 결과물이 있으면 반드시 참고 예시를 첨부해야 합니다. 예시 없으면 AI 자체 기준 결과물이 나오고 수정을 반복하게 됩니다.

##### **케이스 9. 그룹-B 수정 시 A,C까지 건드림 → 전체 롤백**

관련 방법론: 그룹 수정법 A-B-C

group-B의 데이터 처리 형식을 변경해야 했습니다. 변경 범위가 B에만 해당되는 내용이었습니다. 근데 요청을 이렇게 했습니다.

AI가 group-A의 멀쩡한 코드까지 건드렸습니다. group-C도 불필요하게 수정됐습니다. 원래 잘 작동하던 A와 C에 추가 오류가 생겼습니다. A, B, C 세 그룹이 모두 문제가 되어서 전체 롤백을 해야 하는 상황이 됐습니다.

올바른 방법은 MECE 원칙을 그대로 적용하는 것이었습니다.

•          ① B 폴더 MD 수정 → 코드 삭제 → 재생성

•          ② A 폴더는 마지막 Step 출력부만 확인 (B가 받을 수 있는 형식인지만 체크)

•          ③ C 폴더는 첫 Step 입력부만 확인 (B의 새 출력 형식을 읽을 수 있는지만 체크)

•          ④ hook-master.md에서 A→B, B→C 연결 업데이트

 연결 인터페이스 파일명만 맞춰줬습니다.

 그룹 수정 시 "전체 프로젝트 수정" 요청은 절대 하면 안 됩니다. 해당 그룹만 지정하고 인접 그룹은 연결부만 확인하면 됩니다. MECE 설계 덕분에 B 내부가 변해도 A, C 내부는 항상 안전합니다.

##### **케이스 10. 수정 후 MD 업데이트 안 함 → 2주 후 싱크 불일치**

관련 방법론: MD 라이프사이클 (MD로 시작해서 MD로 끝난다)

step2 로직을 수정했습니다. 코드는 잘 작동했습니다. "끝났다"하고 넘어갔습니다. SPEC.md와 수정가이드 업데이트를 깜빡했습니다.

2주 후였습니다. 다른 팀원이 step2를 다시 수정해야 했습니다. SPEC.md를 펼쳤더니 2주 전 버전입니다. 그 SPEC.md 기준으로 수정 요청을 했더니 현재 코드와 충돌이 났습니다. 수정가이드도 예전 정보라 다른 팀원이 혼란스러워했습니다.

해결 방법은 수정 완료 후 Leader한테 이 한 마디를 습관화하는 것입니다.

 Leader Agent가 전체 MD를 최신화해줍니다. 5분도 안 걸립니다.

 MD를 업데이트하지 않으면 MD와 코드 싱크가 안 맞아서 다음 수정 때 반드시 문제가 생깁니다. 수정의 마지막 단계는 항상 "MD 업데이트 요청"입니다. MD로 시작해서 MD로 끝나야 순환이 완성됩니다.

#### **마무리 - 8가지 핵심 교훈**

자~ 이렇게 파트 1 방법론 14개 챕터, 파트 2 시행착오 10가지 케이스까지 전부 다 다뤘습니다.

처음에 말씀드렸던 것처럼, 이 방법론은 저희가 직접 틀려보면서 만든 겁니다. 설계 없이 시작했다가 3시간 날리고, 파일명 하나 바꿨다가 연쇄 오류 터지고, SPEC.md 300줄 써놨다가 AI가 중간 내용 누락하고... 이 삽질들이 전부 방법론의 재료가 됐습니다.

마지막으로 핵심 8가지 교훈만 다시 한번 정리하고 마무리하겠습니다.

코드 실력보다 설계 능력이 결과물을 결정합니다. 설계 30분이 삽질 3시간을 막습니다. 5계층 구조를 반드시 갖추고 시작하세요.

각 Step과 그룹이 겹치지 않고 전체를 빠짐없이 커버해야 합니다. 이게 안 되면 수정할 때마다 전체가 흔들립니다. 사업 설계랑 정확히 같은 원리입니다.

파일명이 곧 Step 간 계약입니다. 이게 없으면 연결 오류가 반드시 터집니다. SPEC.md에서 중요도 별 다섯 개짜리 항목입니다.

3개 이상 연쇄 수정이면 망설이지 말고 전면 재시작해야 합니다. MD만 남기고 코드 밀고 새로 짜는 게 디버깅보다 10배 빠릅니다. 코드는 버려도 됩니다. MD만 살아있으면 언제든 재생성됩니다.

B를 변경할 때 A와 C는 연결부만 확인하면 됩니다. MECE로 나눠 놓았기 때문에 B 내부가 아무리 바뀌어도 A, C 내부 로직은 안전합니다. "전체 수정"은 절대 하지 마세요.

터미널 탭 여러 개에서 Claude Code를 동시 실행하면 됩니다. 별도 빌드나 특수 설정 없이 탭만 열면 됩니다. 설계는 PC 버전, 개발은 터미널 버전으로 역할을 나누면 최적입니다.

AI에게 예시를 주면 역설계해서 그대로 만들어줍니다. 모델과 API를 명시하지 않으면 저품질 모델이 자동 선택됩니다. 둘 다 SPEC.md에 반드시 써야 합니다.

수정 후 MD 업데이트 요청이 마지막 단계입니다. 이 순환이 완성되어야 방법론이 살아있습니다. MD와 코드 싱크가 안 맞으면 다음 수정 때 반드시 문제가 생깁니다.

이 8가지, 한번에 다 적용하려고 하지 않으셔도 됩니다. 지금 당장 하나만 챙겨가세요. 설계도.md 하나 만들고 시작하는 것. 그것만 해도 이전과 완전히 달라집니다.

저희 CCFM에서 이 방법론으로 실제로 운영하고 있는 자동화 툴들이 있습니다. 카카오톡 봇, 광고 크리에이티브 생성 시스템, 영상 제작 파이프라인까지. 전부 비개발자가 이 방법론으로 만들고 유지보수하고 있습니다.
